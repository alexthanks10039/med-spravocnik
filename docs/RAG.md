# RAG-библиотека для Codex

RAG-библиотека хранится в `src/rag/library.ts`. Она отделяет разные типы знаний, чтобы коммерческие решения, инженерные инструкции и медицинский контент не смешивались в одном контексте.

## Пространства

### `commercial`

Позиционирование, аудитория, тарифы, продуктовые ограничения, партнёрства и коммерческие правила. Здесь не должны храниться медицинские рекомендации или секреты.

### `development`

Архитектура, API-контракты, инфраструктура, соглашения кода, сценарии запуска и технические решения. Документы должны соответствовать текущему состоянию репозитория.

### `content`

Таксономия, редакционная политика, медицинские справочные материалы, требования к источникам и правила пересмотра. Любое клиническое утверждение должно иметь владельца, источник и дату проверки до production-публикации.

## Формат документа

```ts
type RagDocument = {
  id: string;
  space: 'commercial' | 'development' | 'content';
  title: string;
  summary: string;
  tags: string[];
  source: string;
  updatedAt: string;
};
```

`id` должен быть стабильным и уникальным. `summary` хранит самодостаточный фрагмент знания, `tags` улучшают поиск, `source` показывает происхождение, `updatedAt` использует ISO-дату `YYYY-MM-DD`.

## Добавление документа

1. Выберите только одно пространство.
2. Убедитесь, что документ не дублирует существующий.
3. Добавьте объект в соответствующий массив.
4. Используйте фактическое название продукта и актуальные endpoints.
5. Для медицинского содержания укажите проверяемый источник вместо `internal`.
6. Проверьте результат через `GET /api/rag?q=<query>&space=<space>`.

## Поиск

Текущий MVP выполняет регистронезависимый substring-поиск по заголовку, summary и тегам. Он не создаёт embeddings и не использует vector database.

Для следующего этапа рекомендуются chunking по смысловым блокам, метаданные доступа, PostgreSQL + pgvector, hybrid search и reranking. Коммерческие документы должны иметь отдельные права доступа от публичного медицинского контента.

## Контроль качества

- удалять устаревшие документы или обновлять `updatedAt`;
- не помещать токены, пароли, персональные и медицинские данные пользователей;
- различать подтверждённые факты и продуктовые гипотезы;
- для медицинского контента хранить источник, редактора и срок следующего пересмотра;
- проверять, что ответ RAG не превращает справочный материал в индивидуальное назначение.
